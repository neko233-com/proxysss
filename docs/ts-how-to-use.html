proxysss 的 TypeScript 运行时是内嵌 QuickJS，不依赖 node、deno 或 tsc。它的目标是给你业务感知、亲和选路、实验逻辑和插件扩展点，而不是替代 YAML 主配置面。

First decision

先判断这件事该不该交给脚本。

如果规则脱离业务上下文后仍然成立，它大概率应该写在 YAML；只有业务感知、租户感知、player_id 感知和特殊上游选择，才适合进入 TS。

优先用 YAML 适合用 TS
固定 host/path 反代、静态文件、WebDAV、FTP、TCP、UDP、TLS、健康检查、缓存、限流、访问控制。 业务相关 header 注入、player_id / tenant / device 维度选路、亲和路由、灰度、实验逻辑、特殊 API 兼容层。
Setup

最小可用脚本应该足够小，先验证加载链路。

推荐布局是 `gateway.ts` 作为主入口，`plugins/*.ts` 放可选插件，`proxysss-script.d.ts` 提供类型提示。

第一个最小插件 
/// <reference path="./proxysss-script.d.ts" />

const plugin: ProxysssPlugin = {
  name: "gateway",
  priority: -1000000,
  enabled: true,

  access(message, current) {
    if (message.ctx.path === "/healthz") {
      return { upstream: "proxysss://healthz" };
    }
    return current;
  },
};

export default plugin;
Hooks

最常用的几个 hook，一次把职责边界讲清。

hook 应该保持同步、快速、可解释。它是扩展点，不是长期业务 worker。

HTTP 路由阶段

`access`

适合 host/path 级决策，比如动态改写 path、插 header，或者把很少量特殊 path 重定向到内建上游。

access(message, current) {
  if (message.ctx.host === "api.example.com" &&
      message.ctx.path.startsWith("/tenant-a/")) {
    return {
      ...current,
      rewrite_path: message.ctx.path.replace("/tenant-a", "/api"),
      set_headers: {
        "x-tenant-id": "tenant-a",
      },
    };
  }
  return current;
}
上游选择阶段

`balancer`

路由已经命中，只想改上游选择或追加 header 时最合适，典型是灰度、亲和和流量打标。

balancer(message, current) {
  const user = message.ctx.headers["x-user-id"];
  if (user && Number(user) % 10 === 0) {
    return {
      ...current,
      upstream: "http://10.0.0.12:8080",
      set_headers: {
        "x-release-bucket": "canary",
      },
    };
  }
  return current;
}
TCP / UDP 预读

`preread`

适合根据 listener、player_id、首包预览做非 HTTP 流量落点选择，比如游戏分区和设备亲和。

preread(message, current) {
  if (message.listener === "game-tcp" && message.ctx.player_id) {
    const shard = Number(message.ctx.player_id) % 2;
    return {
      ...current,
      upstream: shard === 0 ? "10.0.1.10:7000" : "10.0.1.11:7000",
    };
  }
  return current;
}
观测

`log`、`init_worker`、`onDispose`

`log` 适合观测和统计；`init_worker` 适合初始化插件状态；`onDispose` 适合卸载或热重载时清理资源。它们都不适合做慢操作。

Fields

最常用字段不要拍脑袋猜，先对照类型声明。

HTTP 和流式 listener 的字段不同。想用哪个字段，先对照 `proxysss-script.d.ts`，不要发明不存在的属性名。

HTTP TCP / UDP
`message.ctx.host`、`method`、`path`、`query`、`remote_addr`、`player_id`、`headers` `message.listener`、`message.ctx.protocol`、`remote_addr`、`player_id`、`first_packet_preview`、`payload_len`
Return values

返回值的理解顺序：先 upstream，再 path，再 header。

最常见的写法是覆盖 `upstream`，其次是 `rewrite_path`，最后才是 `set_headers` / `strip_headers` 这类附加动作。

最小可用 
return { upstream: "http://127.0.0.1:8080" };
带 upstream pool 
return {
  upstream: "http://127.0.0.1:8080",
  upstreams: [
    "http://127.0.0.1:8080",
    "http://127.0.0.1:8081",
  ],
};
带 path 和 header 改写 
return {
  upstream: "http://127.0.0.1:9000",
  rewrite_path: "/internal/api",
  set_headers: {
    "x-proxysss-plugin": "custom-business-route",
    "x-tenant-id": "tenant-a",
  },
  strip_headers: ["x-remove-me"],
};
Mistakes

写 TS 时最容易走歪的四个方向

这些错误的共同点是：把脚本当成主配置面、把 QuickJS 当 Node、把 hook 当 worker。

误区 1

把普通反向代理全搬进 TS

YAML 更稳定、更快，也更容易给团队和 agent 看懂。普通网关规则优先留在 YAML。

误区 2

把 QuickJS 当 Node.js

不要假设有 `fetch`、`fs`、`process` 或 `node:*` 模块。这是内嵌 QuickJS，不是 Node 运行时。

误区 3

在 hook 里做重逻辑或慢逻辑

hook 必须保持同步和快速。慢逻辑会直接伤害热路径和可预测性。

误区 4

用脚本掩盖配置面缺口

如果某个能力应该是通用网关能力，就应该推动 YAML / Rust 本体，而不是长期用脚本打补丁。
