Beginner path

如果你是新手，先把一条链路跑通。

这不是“先背概念再动手”的文档。你应该先有一个能工作的配置，然后再知道为什么它工作。

5 分钟可用

第一个反向代理

你只有一个站点，要把 `app.example.com` 代理到本机 9000 端口。

http:
  plain_bind: 0.0.0.0:80

services:
  reverse_proxy:
    routes:
      - name: app
        match:
          hosts: ["app.example.com"]
          path_prefix: "/"
        upstreams:
          - url: "http://127.0.0.1:9000"
正式上线

只填域名就给 WebSocket 加 WSS

当 80 / 443 能直连公网时，只填写域名即可自动签正式证书；不需要证书脚本、DNS API 或邮箱。

http:
  plain_bind: 0.0.0.0:80
  tls_bind: 0.0.0.0:443
  tls:
    auto_https:
      domains: [wss.example.com]

Static + API

官网静态站点 + API 反向代理

官网直接从磁盘分发，HTML/CSS/JS/图片/字体/音视频支持缓存、大文件流式发送和 Range 断点续传；API 单独走后端服务，便于缓存和排障。

services:
  static_sites:
    - name: homepage
      hosts: ["www.example.com"]
      root_dir: "./www"
      index_files: ["index.html"]

  reverse_proxy:
    routes:
      - name: api
        match:
          hosts: ["api.example.com"]
          path_prefix: "/"
        upstreams:
          - url: "http://127.0.0.1:8080"

AI / SSE

给 LLM 流式输出做网关

面向 OpenAI-compatible / New API / SSE 长响应，强调小包 flush 和无副作用性能优化。

runtime:
  performance:
    enabled: true
    traffic_profile: small

services:
  ai_proxy:
    routes:
      - name: llm-edge
        match:
          hosts: ["ai.example.com"]
          path_prefix: "/v1"
        upstreams:
          - url: "https://api.openai.com"
        transport:
          flush_interval_ms: 0
          tcp_nodelay: true
Weighted cluster

权重 + 健康检查 + 缓存 + 限流

给正式 API 集群用的骨架，不是演示配置。适合上线前进一步叠加观察和压测。

services:
  reverse_proxy:
    routes:
      - name: api-cluster
        match:
          hosts: ["api.example.com"]
          path_prefix: "/"
        upstreams:
          - url: "http://10.0.0.11:8080"
            weight: 3
          - url: "http://10.0.0.12:8080"
            weight: 2
        load_balance:
          strategy: weighted_round_robin
          active_health:
            path: "/healthz"
            interval_secs: 5
            timeout_ms: 1200
Game / Realtime

游戏 TCP、UDP、KCP、QCP

HTTP 以外的边缘网关能力也要明确可配，不需要把实时服务硬塞进反向代理语义里。

tcp:
  listeners:
    - name: game-tcp
      bind: 0.0.0.0:7000
      nodelay: true
      routes:
        - name: zone-a
          upstreams:
            - addr: "10.0.1.10:7000"

udp:
  listeners:
    - name: game-udp
      bind: 0.0.0.0:7001
      session_ttl_secs: 45
      max_associations: 150000
MQTT / IoT

MQTT TCP、MQTT over WebSocket、CoAP-style UDP

proxysss 是设备和 broker 前面的边缘网关，不是假装自己是 broker。

services:
  reverse_proxy:
    routes:
      - name: mqtt-ws
        match:
          hosts: ["iot.example.com"]
          path_prefix: "/mqtt"
        upstreams:
          - url: "http://127.0.0.1:8083"

tcp:
  listeners:
    - name: mqtt-tcp

Wildcard TLS

泛域名证书 + DNS-01

做 `*.example.com` 或多节点证书管理时，直接走内建 `acme_managed` 的 DNS provider 工厂。

http:
  tls:
    mode: acme_managed
    acme:
      enabled: true
      email: "ops@example.com"
      challenge: dns01
      dns:
        provider: cloudflare
        credentials:
          api_token: "REDACTED"
Expert path

如果你是高手，先看能力面边界，再决定配置落点。

这张表不是营销词汇表，而是“应该配到哪里”的决策速查。

配置面 适合什么 不要拿来做什么
services.reverse_proxy.routes普通 HTTP / API / WebSocket / SSE 大多数网站、后台、流式 API 都从这里开始。 不要用它去伪装 TCP / UDP 原生协议。
services.ai_proxyOpenAI-compatible / New API 需要更明确的 AI 网关语义、上游池、SSE 传输策略。 不要把所有 HTTP 都硬塞进 AI proxy。
services.domain_routes站点级编排 按域名把静态、API、WebDAV 等能力拼成一个站点。 不要把它当成唯一实际执行面。
services.static_sites静态资源 / CDN 回源 HTML、CSS、JS、图片、字体、音视频、大文件下载、Range 断点续传。 不要把业务鉴权写进静态文件层，复杂策略放脚本或上游。
services.service_discoveryConsul / etcd / Nacos 把注册中心服务映射到 HTTP、TCP、UDP 上游池的控制面声明。 不要让每个热路径请求临时查注册中心。
tcp.listenersTCP 原生流量 数据库、游戏长连接、MQTT TCP、TLS passthrough/SNI 场景。 不要期待 HTTP 头级别策略。
udp.listenersUDP、KCP、QCP、CoAP-style 实时设备、游戏、会话型 UDP 网关。 不要把它包装成应用终端本身。
