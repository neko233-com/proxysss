这页到底在讲什么

这是“官方 Linux 全协议 mixed-load benchmark”的人类可读解释页，但现在不只解释规则，也直接展示历史跑分基线。这样你打开页面时，先看到真实 Ubuntu 24 / Docker 对标 nginx 的结果，再去理解为什么后续优化必须按 mixed-load 口径继续压测。

它适合回答：

当前主干在 Linux 上，static、reverse proxy、New API/SSE、WebSocket、TCP、UDP 这些 nginx 可公平对标场景一起跑时，到底有没有一起站住，以及历史上到底跑到了什么水平。KCP/QCP 作为 proxysss 独立 UDP listener 能力单独验证。

它不适合回答：

不能只拿一个 static small 的快照，就说发布已经安全，或者说所有协议都自动跟着变快。

官方 Linux benchmark 现在比较哪些协议

默认 mixed matrix 不是单协议测试，而是一整个真实网关波次。下面这些场景要一起看，而不是挑其中一个最漂亮的图：

场景 协议 / 关注点 为什么必须一起比
static-small / static-large / cdn-hot-update / https-static-small HTTP / HTTPS 静态分发 验证小文件热路径、批量传输、TLS 和 cache/sendfile 取舍是不是一起稳定。
reverse-proxy / new-api-sse HTTP reverse proxy / AI streaming / SSE 验证 keepalive、flush、forward headers、AI metadata 和低分配快路径是不是一起正确。
websocket-long-connection / game-long-connection / tcp-stream WebSocket / 长连接 TCP / stream proxy 验证长连接时延、连接 churn、stream relay 和 realtime 公平比值底线。
udp-stream UDP 验证会话 TTL、关联上限、缓冲、丢包和 realtime 稳定性；KCP 与 QCP 作为 proxysss 独立 UDP listener 能力单独验证，不进入默认 nginx head-to-head gate。

GitHub Actions 工件现在应该长什么样

Linux 官方工件：

`benchmark-results-linux` 应该打包 `.benchmark/runs/all-scenarios/results.json`、`summary.md`、`summary.html`，而不是只塞一个 static quick report。

Windows smoke 工件：

可以继续保留 throughput smoke，用来发现 bench 命令和构建链有没有炸，但它不是官方发布性能口径。

团队约束很明确：性能优化必须是 no-side-effect optimization。如果某个局部路径更快了，但兄弟场景更慢、更脆、观测性更差，那不算成功。

GitHub-hosted Linux runner 和专门 Linux 主机不是一回事

GitHub-hosted runner：

现在比较 nginx 可公平对标的主要网关协议；hosted runner 上的 UDP 噪声仍只适合看趋势，不替代专门 Linux 主机结论。

专门调优过的 Linux 主机：

这里才是 `realtime UDP` 的硬门槛环境。KCP/QCP 走 proxysss 独立 UDP listener 能力验证，不拿 nginx 做协议语义对照。

Docker / WSL2 容器如果缺 `/proc/sys/net/core/rmem_max`，脚本会自动把 UDP error tolerance 放宽到 `proxysss <= nginx + 16` 并在 summary 明示；真 Linux 主机默认仍是 `+4`。

正式发布为什么还要 mixed-load

正式 Linux 发布口径仍然是 mixed-load，而不是单场景快照。要一起看这些路径：

HTTP 族

static-small、static-large、CDN hot-update、HTTP reverse proxy、New API/SSE、HTTPS static。

长连接 / 实时族

WebSocket、game TCP、generic TCP、UDP。

如果你在调 SSE 或 HTTP/2，最终也必须证明它没有把 static、reverse proxy、TCP、UDP 一起拖慢。

怎么复现正式 Linux benchmark

正式发布前请先把 Linux host 调优到位，再跑 mixed matrix：

proxysss tune linux --apply --profile latency --max-connections 200000
PROXY_BIN=/usr/local/bin/proxysss QUICK=1 DURATION_SECS=12 MIXED_MATRIX=1 \
  bash scripts/benchmark-all-scenarios.sh

如果你只想定位某个局部热点，单场景 quick throughput 仍然有价值；但它现在只能算诊断工具，不能再替代官方 Linux mixed-load benchmark。

和这次 SSE / HTTP2 优化的关系

这次不是简单追一个 SSE 数字，而是要求快路径继续保留默认 `X-Forwarded-*` / `Forwarded` 语义，以及 `proxysss-ai-*` metadata header，同时把 HTTP/2 默认调优抬高。也就是说，优化必须建立在“默认行为不倒退”的前提上。

正式发布请继续坚持这条线：先 Linux 调优，再 mixed-load benchmark，再看 CI / benchmark / deploy / release 全绿。历史真机基线已经摆在这里，后续优化必须整体对比，而不是拿单场景截图替代发布标准。
