这页到底在讲什么

这是 Linux benchmark 的人类可读解释页。v1.3.5 之后，页面先展示当前 UDP fast path 的专项结果，再展示旧 mixed-load 历史基线。这样打开页面时不会把历史 `udp-stream 0.855x` 误读成这次 UDP 优化后的性能。

它适合回答：

当前 UDP fast path 是否已经明显超过 nginx，以及历史 mixed-load 基线里 static、reverse proxy、generic SSE、WebSocket、TCP、UDP 这些 nginx 可公平对标场景整体跑到过什么水平。

它不适合回答：

不能把旧 mixed-load 表里的单行历史数字，当成 v1.3.5 UDP fast path 的当前结果；也不能只拿一个单场景快照替代完整 mixed-load。

单网关 WSS 隔离 Docker 验证

Rust 游戏网关先跑 scripts/benchmark-websocket-production-gate.sh：nginx 和 proxysss 都固定为同一组 4 CPU 网关容器，4 个回源和多个客户端处在独立 cgroup 与网络命名空间。每个候选都跑相同的 WSS 活跃回显吞吐、p50/p95/p99 延迟，以及 20k idle WSS 容量，脚本自动写出百分比对比和网关资源快照；内存默认记录而不是用任意阈值淘汰，只有声明真实生产预算时才传 Docker/systemd 内存上限。

proxysss tune linux --apply
bash scripts/benchmark-websocket-production-gate.sh

默认容量用 5 个客户端 IP、4 个后端 IP:port 避免 TCP 四元组上限；推荐压测机至少 16 CPU / 32 GiB，脚本会拒绝 cpuset 重叠或默认包络之外的主机。WSS 使用生成的自签名 fixture，只有压测客户端使用 --insecure，因此验证的是 TLS/WSS 数据路径而不是证书信任。Docker 隔离可以排除同容器/同进程干扰，但仍共享一个 Linux 内核；生产发布必须再用 scripts/benchmark-cross-host-wss.sh 在独立网关、回源、压测主机复跑：它验证 machine-id 角色独立，并用 systemd cgroup 强制 4 CPU / 300k nofile，保存 SHA、cgroup memory current/peak、每连接成本、主机指纹、nginx 构建信息与原始结果。只有声明实际生产内存预算时才设置 GATEWAY_MEMORY_MAX。

需要规模证据时，用 scripts/benchmark-cross-host-scale-matrix.sh 一条命令按 1x/2x/4x 复跑三机 WSS：默认 active WSS 为 4096、8192、16384，idle 容量始终为 20k。这验证规模增长而不把容量人为推到 100k。

若历史 benchmark Docker network 占用默认子网，可传未占用的 BENCH_SUBNET=172.31.0.0/16；WSS 脚本会将 nginx/proxysss/backend/client 的整套 Latin-square 地址同步派生到该子网，不能只换 network 却继续使用旧 172.30.* 地址。受限 Linux 控制器没有合适 Go 工具链时，可从本仓库交叉构建同架构 Go 原生 helper 并传 PREBUILT_BENCH_HELPER=/opt/benchmark-helper；脚本只接受可执行的本地 helper，不会改用 Python/Node。

正式 tag 发布还必须带版本化 performance-evidence/vX.Y.Z.json：schema v2 不接受“已胜出”的布尔声明，而是逐场景固化两边的 ops/s、p50/p95/p99 和 errors；验证器要求 proxysss 吞吐严格更高、三个百分位严格更低、两边 0 错误。全场景隔离必须覆盖 static、CDN hot-update、HTTPS、reverse proxy、generic SSE、WebSocket、game TCP、TCP、UDP 与透明 QCP edge forwarding；容量还要记录两边 opened/failed、open rate 和握手 p50/p95/p99，并限制在 1k-50k 的可复现包络。每个 1x/2x/4x 的全场景隔离与三机 WSS 运行还要记录角色隔离、proxysss/nginx 各自的内存 current/peak/每连接成本及无持续增长；proxysss 的任一项不得超过 nginx 2 倍，并且要带 saturation/equal-load/capacity 原始工件的 URI 和 SHA-256；三机 WSS 必须给出三个不同的 machine-id hash。release 工作流验证清单的 tag/commit 并将其作为 release asset；这会阻止缺少真实 Linux 证据时误发版本，但原始工件仍必须人工审计。

官方 Linux benchmark 现在比较哪些协议

默认 mixed matrix 不是单协议测试，而是一整个真实网关波次。下面这些场景要一起看，而不是挑其中一个最漂亮的图：

场景 协议 / 关注点 为什么必须一起比
static-small / static-large / cdn-hot-update / https-static-small HTTP / HTTPS 静态分发 验证小文件热路径、批量传输、TLS 和 cache/sendfile 取舍是不是一起稳定。
reverse-proxy / generic-sse HTTP reverse proxy / generic SSE streaming 验证 keepalive、flush、forward headers 和低分配快路径是不是一起正确。
websocket-long-connection / game-long-connection / tcp-stream WebSocket / 长连接 TCP / stream proxy 验证长连接时延、连接 churn、stream relay 和 realtime 公平比值底线。
udp-stream UDP 验证会话 TTL、关联上限、缓冲、丢包和 realtime 稳定性；KCP 与 QCP 作为 proxysss 独立 UDP listener 能力保留，不进入当前性能 benchmark 矩阵。

20k WebSocket 容量与消息吞吐不是同一个指标

ops/s 衡量已建立连接上的消息回显轮次，不是并发连接数。容量验证应使用原生的只建连保持模式；它会报告请求、成功、失败、建连速率与握手 p50/p95/p99，而不会让每条连接持续刷消息。

proxysss bench websocket \
  --url ws://gateway.example.com/gateway/ws \
  --connections 20000 \
  --hold-connections \
  --connect-workers 128 \
  --connect-timeout-ms 10000 \
  --connect-retries 4 \
  --duration-secs 30
TCP 四元组上限

一个 IPv4 源地址到一个后端 IP:port 的临时端口数量有限，默认 Linux 通常只有约 2.8 万。默认 20k 容量测试仍使用多个压测源地址，并把 WebSocket upstream 分到多个不同后端 IP:port（或使用代理源 IP 池），避免端口限制被误判成网关失败。nginx 与 proxysss 都受此限制。

proxysss 分流语义

在 upstreams 中声明多后端后，未启用玩家亲和时 WebSocket 快路径 round-robin 分流；启用 affinity 后使用 Rendezvous 粘性。容量通过后，仍必须跑活跃消息延迟与完整 mixed-load，不能用 idle 连接数代替实时性能。

CI 和 benchmark 的边界

默认 CI：

现在只做六个平台 release binary 构建、打包、上传 artifact；禁止自动或手动 workflow 运行 test、smoke 或性能 benchmark。

手动 benchmark：

scripts/benchmark-ubuntu24-amd64-docker.sh 是本机或原生 Docker 严格 1x/2x/4x+QCP 入口；它硬校验 Ubuntu 24.04 x86_64 容器并记录 native/emulated amd64；arm64 daemon 通过 Zig + cargo-zigbuild 本机交叉编译 release ELF，再在 Ubuntu 24 amd64 容器执行校验，避免 QEMU 编译进入反馈时间。底层 scripts/benchmark-all-scenarios-isolated.sh 把 gateway、backend、client 固定到互不重叠的 cpuset；每尺度常驻一个 backend 与两边 gateway、pause 非被测方，每个 wave 的单一 client 容器内运行 11 个独立协议进程并共享同一绝对开始时间；saturation client 使用完整 client cpuset；fixed-rate equal-load 的小包 client 各用 1 个 Tokio I/O worker、static-large 用 2 个，避免发生器自身 thread oversubscription；迟到 tick 在单 in-flight 边界内补齐。mixed saturation 与 equal-offered-load 默认在每个 1x/2x/4x 尺度对每个 gateway/phase 各跑 1 个同步 2 秒样本，使完整反馈目标约 1 分钟；显式多轮时才取中位数；equal-load 使用双方较慢 saturation 的 25%，为 11 个并行 generator 保留 latency headroom，要求双方完成至少 98% target，proxysss 还必须逐项严格胜出吞吐和 p50/p95/p99。scripts/benchmark-all-scenarios.sh 保留为旧 mixed 诊断入口，SCENARIO_FILTER=udp-stream 只用于定位 UDP fast path。

团队约束很明确：性能优化必须是 no-side-effect optimization。如果某个局部路径更快了，但兄弟场景更慢、更脆、观测性更差，那不算成功。

GitHub-hosted Linux runner 和专门 Linux 主机不是一回事

GitHub-hosted runner：

禁止用于性能诊断或正式压测；Actions 只负责打包，诊断在本机 Ubuntu 24 amd64 Docker 运行，原生生产证据来自专机。

专门调优过的 Linux 主机：

这里才是 `realtime UDP` 的硬门槛环境。KCP/QCP 走 proxysss 独立 UDP listener 能力验证，不进入当前 nginx 性能矩阵。

Docker / WSL2 容器如果缺 `/proc/sys/net/core/rmem_max`，脚本会自动把 UDP error tolerance 放宽到 `proxysss <= nginx + 16` 并在 summary 明示；真 Linux 主机默认仍是 `+4`。

正式发布为什么还要 mixed-load

正式 Linux 发布口径仍然是 mixed-load，而不是单场景快照。要一起看这些路径：

HTTP 族

static-small、static-large、CDN hot-update、HTTP reverse proxy、generic SSE、HTTPS static。

长连接 / 实时族

WebSocket、game TCP、generic TCP、UDP。

如果你在调 SSE 或 HTTP/2，最终也必须证明它没有把 static、reverse proxy、TCP、UDP 一起拖慢。

怎么复现正式 Linux benchmark

正式发布前请先把 Linux host 调优到位，再跑 mixed matrix：

proxysss tune linux --apply --profile latency --max-connections 200000
bash scripts/benchmark-ubuntu24-amd64-docker.sh
PROXY_BIN=/usr/local/bin/proxysss QUICK=1 DURATION_SECS=12 MIXED_MATRIX=1 \
  bash scripts/benchmark-all-scenarios.sh

如果你只想定位某个局部热点，单场景 quick throughput 仍然有价值；但它现在只能算诊断工具，不能再替代官方 Linux mixed-load benchmark。

和这次 SSE / HTTP2 优化的关系

这次不是简单追一个 SSE 数字，而是要求快路径继续保留默认 `X-Forwarded-*` / `Forwarded` 语义，以及 `proxysss-ai-*` metadata header，同时把 HTTP/2 默认调优抬高。也就是说，优化必须建立在“默认行为不倒退”的前提上。

GitHub Actions 只负责打包且禁止性能压测；性能判断必须直连 Linux 专机运行 mixed-load 和必要的单场景诊断。v1.3.5 的 UDP fast path 当前专项结果是 4.045x，历史 mixed-load 表只作为整体基线保留。
