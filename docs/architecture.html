Click a protocol and watch the traffic idea change.

For a freshman: every protocol is just an agreement about how two computers talk. proxysss does not need to understand every business meaning inside the bytes. It only needs enough edge knowledge to accept, protect, route, tunnel, and observe the traffic.

HTTP reverse proxy

A browser sends a request with a host and path. proxysss matches the route, forwards to an upstream, and streams the response back.

request

Think of HTTP like ordering food at a counter: "Host is this restaurant, path is this dish, headers are small notes." proxysss reads the order slip and sends it to the right kitchen.

Data path

The gateway is a pipeline, not magic.

A request enters a listener, goes through safety and routing decisions, then becomes either a proxied HTTP exchange, a bidirectional stream, a static/WebDAV/FTP operation, or an admin/docs response.

1. Listen

Ports 80, 443, 7777, TCP, UDP, FTP, and custom binds accept connections or datagrams with explicit TCP backlog and default-on runtime.performance tuning. On Linux, plain HTTP, TLS HTTP, TCP stream, and UDP listener binds fan out across CPU-adaptive SO_REUSEPORT workers when performance mode is enabled. Startup detects the OS version, applies portable Linux socket tuning, enables the Ubuntu 24.x extreme policy when supported, and logs any downgrade reason.

2. Understand enough

HTTP gets host/path/header parsing. TCP and UDP stay mostly transparent. TLS uses SNI and ALPN where needed.

3. Apply policy

Access control, DDoS windows, rate limits, cache, compression, active health, and maintenance state shape the route.

4. Proxy or serve

proxysss forwards to upstream pools with separate hot paths instead of one global tuning profile. HTTP uses a Hyper fast path for general routes and raw plain-HTTP reverse/SSE/WebSocket lanes for simple no-policy routes. Raw HTTP keepalive pooling uses a lock-free bounded queue instead of a shared mutex. Static delivery uses a bounded cache with mmap-backed objects where supported, plus a traffic-profile-aware Linux fast lane: default small mode favors cached hot files and low-latency HTTP2/SSE/WebSocket/TCP/UDP feedback, bulk mode favors sendfile/zero-copy large transfers, and balanced mode prepares both. Config load preloads eligible index/top-level static files or sendfile descriptors according to runtime.performance.traffic_profile. TCP stream listeners run on a dedicated proxysss-stream runtime in performance mode, fan out accept loops with CPU-adaptive SO_REUSEPORT workers on Linux, and bypass the generic upstream planner for single-upstream direct routes when scripts, affinity, active health, and passive health are disabled. UDP listeners in performance mode also fan out datagram receive loops with SO_REUSEPORT, share the global association table, and keep worker-local association cache entries so established datagrams avoid global lookups and only refresh the global TTL timestamp once per second. Stream relays choose independent profiles: game/MQTT/tool long connections use small relay buffers and parallel one-way pumps, while explicit bulk/file stream protocols may use Linux splice zero-copy when runtime performance tuning is enabled.

Single config

YAML is the lab notebook.

nginx spreads power across directives and includes. proxysss keeps the full edge shape in one YAML file so people and agents can inspect it quickly.

services:
  domain_routes:
    - name: app
      domains: [example.com]
      path_prefix: /
      upstream: http://127.0.0.1:9000
      forward_headers: false

  ai_proxy:
    enabled: true
    routes:
      - name: new-api
        provider: new-api
        match_host: ai.example.com
        upstream: http://127.0.0.1:3000
tcp:
  listeners:
    - name: mqtt
      bind: 0.0.0.0:1883
      protocol: mqtt
      nodelay: true
      upstreams:
        - 127.0.0.1:18831

udp:
  listeners:
    - name: game-kcp
      bind: 0.0.0.0:7001
      protocol: kcp
      max_associations: 262144
What can improve next

Stable gateways keep earning trust.

The current design already covers the nginx-replacement surface. The next wins are deeper measurement, more protocol test fixtures, and sharper production runbooks.

Benchmark matrix

Add repeatable Linux CI/staging runs for TLS, HTTP/2, HTTP/3, WebSocket, TCP, UDP, MQTT, FTP, WebDAV, cache, and AI streaming. Production performance validation is Linux-only and must use the mixed matrix by default: enable the realistic multi-feature gateway shape and run CDN/static, HTTPS static, reverse proxy, New API/SSE, WebSocket long connections, game/TCP streams, generic TCP stream, UDP stream, and KCP-style UDP concurrently per gateway. The default critical gate is fairness-adjusted: aggregate throughput plus WebSocket and game/realtime TCP, UDP, and KCP-style scenarios use a 0.97 proxysss/nginx ratio floor because proxysss carries built-in policy and gateway behavior that nginx usually needs extra modules or configuration to approximate; static/reverse/SSE must stay above the soft floor with low errors; HTTPS static and static-large still run and are reported, but only block releases when explicitly included for TLS/static or bulk-transfer releases. Single-scenario runs are diagnostic only, and strict greater-than-nginx gates are opt-in.

Chaos testing

Inject upstream resets, slow bodies, packet loss, DNS failure, log backpressure, reload races, and certificate renewal failures.

Protocol fixtures

Keep small upstream simulators for MQTT, CoAP-style UDP, FTP active/passive, WebDAV clients, and SSE token streaming.

Kernel tuning guide

Keep Ubuntu 22.04, 24.04, 26.04 LTS and Debian socket backlog, file descriptor, ephemeral port, UDP buffer, QUIC UDP, jemalloc, and systemd restart guidance aligned with proxysss tune linux.

Observability packs

Ship dashboard queries and alert rules for p95 latency, upstream quarantine, dropped UDP associations, watchdog restarts, and ACME renewal.

Plugin safety budgets

Add examples for script timeout, memory budget, hook placement, and how to keep business logic away from byte-copy hot paths.
