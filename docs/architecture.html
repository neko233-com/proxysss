proxysss architecture / human-readable

读架构不是为了崇拜内部实现，而是为了知道请求会怎么走。

这页把 proxysss 的架构拆成“你现在需要知道什么”。新手应该先知道 listener、route、policy、upstream 的顺序；高手应该快速确认脚本 hook 放在哪里、哪些东西能热重载、哪些优化必须按 mixed-load 证明无副作用。

Mental model

先记住 5 个角色，架构就不会看乱。

proxysss 不想把所有东西都折叠成“神秘代理黑盒”。它把每个责任面尽量拆开，而且名字要像人在排障时会说的话。

1

Listener

先接住流量。HTTP / HTTPS / H2 / H3、TCP、UDP、FTP 都有自己的监听入口。

2

Route surface

决定这条流量落到哪个能力面：`reverse_proxy`、`ai_proxy`、`static_sites`、`webdav`、`tcp.listeners`、`udp.listeners`。

3

Policy chain

限流、缓存、压缩、访问控制、健康检查都在这里影响行为，但不应该把数据路径锁死成低性能结构。

4

Extension hooks

TypeScript 只在该出现的地方出现：HTTP 的 `access` / `balancer`，以及 stream 的 `preread` / `log`。

5

Runtime operations

配置加载、warm-up、健康状态、token、热重载边界、watchdog、metrics 一起构成真正能运维的网关。

Listener HTTP / TCP / UDP / FTP 先接流量
→
Route surface 域名、路径、SNI、listener 名称决定能力面
→
Policy chain 缓存、限流、健康检查、压缩、ACL
Extension hooks 只在需要业务感知时介入
→
Upstream / file / service 进入上游池、静态文件、WebDAV 或内建服务
→
Logs / metrics / health 观测、告警、验证、自动化接口
HTTP flow

一条 HTTP 请求怎么走

这是大多数人最先要理解的路径，因为它串起了反向代理、AI SSE、静态站点和脚本扩展。

Step 1

进入 HTTP listener

请求先到 `http.plain_bind` 或 `http.tls_bind`。TLS、HTTP2、HTTP3 是否参与，在这里决定。

Step 2

按 host / path 命中能力面

此时会决定它属于 `reverse_proxy.routes`、`ai_proxy.routes`、`static_sites` 还是 `webdav`。

Step 3

应用策略链

限流、缓存、访问控制、响应策略、主动健康检查等在这里影响行为。

Step 4

必要时进入 TS hook

只有当 YAML 表达不了业务决策时，`access` / `balancer` 才应该介入。

Step 5

发往上游或本地能力面

最终不是去上游池，就是读静态文件、走 WebDAV、返回内建服务结果；静态文件支持 Range/206 断点续传，大文件区间会流式发送。

http:
  plain_bind: 0.0.0.0:80
  tls_bind: 0.0.0.0:443

services:
  reverse_proxy:
    routes:
      - name: app
        match:
          hosts: ["app.example.com"]
          path_prefix: "/"
        upstreams:
          - url: "http://127.0.0.1:9000"

这份最小配置对应的架构路径很直白：`TLS/HTTP listener -> reverse_proxy route match -> upstream connect -> access/error log`。

普通公网 WSS 可把 `http.tls.auto_https.domains` 直接写成域名列表：非空列表会选中内建 managed ACME、正式 Let's Encrypt 和默认 TLS-ALPN-01。临时 `acme-tls/1` 证书、证书持久化、续期和 reload 都由网关处理，不需要外部证书工具、DNS API 或账号邮箱；A/AAAA 必须指向网关且公网 443 可达。邮箱仅用于到期/安全通知；显式 HTTP-01（需 80）、TLS-ALPN-01、DNS-01 与外部 ACME 配置仍兼容。

Stream flow

一条 TCP / UDP 流量怎么走

这里是 proxysss 和“只会讲 HTTP 的文档”最不一样的地方。实时流量不是附属功能，而是一等公民。

TCP

TCP listener

按 listener 名称、SNI、直通规则决定流量去哪里。典型场景是数据库、MQTT TCP、游戏长连接。

UDP

UDP listener

重点不只是转发，还有 `session_ttl_secs` 和 `max_associations` 这类会话治理；KCP 和 QCP 是独立 listener，协议帧都留给各自上游服务。

Hook

`preread` 只做前置选路

如果你真要做 player_id / device 亲和，脚本应该只负责快速决定 upstream，不要把慢逻辑塞进热路径。

tcp:
  listeners:
    - name: game-tcp
      bind: 0.0.0.0:7000
      nodelay: true
      connect_timeout_ms: 1500
      routes:
        - name: zone-a
          upstreams:
            - addr: "10.0.1.10:7000"

udp:
  listeners:
    - name: game-udp
      bind: 0.0.0.0:7001
      protocol: qcp
      session_ttl_secs: 45
      max_associations: 150000
      routes:
        - name: zone-a-udp
          upstreams:
            - addr: "10.0.1.10:7001"

Linux 性能模式下，listener 按 `sched_getaffinity` 检测到的 cpuset 建立 `SO_REUSEPORT` accept worker。plain `ws://`、游戏小帧 TCP 与通用 1 KiB TCP 进入按 cpuset 线性扩展的原生 `epoll` relay：`small` 每 2 个可用 CPU 一个 owner，默认混合负载 `balanced` 与 `bulk` 每 4 个 CPU 一个 owner，为并发 HTTP/TLS/UDP/file 保留 scheduler 预算。reactor worker 少于 cpuset 核数时使用 soft CPU ownership，Linux 通常保持 cache-local，只在 runnable imbalance 时迁移；一核一 owner 才反向硬 pin，避免 2 核/1 owner 固定压住 CPU1。balanced 使用 nice +5，在同核 HTTP/TLS 可运行时有界让出预算，但不再像旧 +12 那样把 realtime wake 延迟到 nginx 之后。每条连接由单一 reactor owner 维护 backpressure 与 half-close；`EPOLLHUP`/`EPOLLRDHUP` 与 `EPOLLIN` 同时出现时先 drain 尾部数据再传播 half-close，避免丢弃最后一帧。无法 handoff 时回退到有界 Tokio relay。默认 balanced 把 plain HTTP、TCP accept 与 UDP recv/association 固定到同一组 per-core Tokio shard；worker index 决定 CPU owner。connection-local data runtime 每 31 次 task poll 检查一次 global injection queue、每 16 次驱动一次 I/O/timer，去掉旧 2/3 间隔的小包 scheduler 浪费，同时保持低于 Tokio 默认值的 I/O interval。TLS 使用 `ceil(cpuset cores / 4)`、nice +3 的独立 work-stealing crypto runtime，在高核主机上线性扩展，同时限制 mixed TLS CPU 份额并避免固定热点核。bulk/file profile 使用 `splice`。达到每 base shard 64 条活跃 TLS 连接后才弹性启动额外 TLS accept shard。

每次 level-triggered readiness 最多连续 drain 8 个 16 KiB read batch，合并已排队 WebSocket/game/TCP 小帧的 epoll/hash dispatch；达到上限就归还 event loop，partial write 立即进入显式 pending/backpressure。普通数据事件只在本端已经观察到 EOF 后才做完整 pair-finished 查询，去掉每帧两端状态查找。

Linux 性能模式下 UDP 使用全核 CPU 自适应 `SO_REUSEPORT` socket fanout、复用有界 UDP buffer pool；balanced 的 recv loop 与 association reader 留在对应的 per-core data shard，与该核 HTTP/TCP accept 共用一个 reactor，既获得全核并行度，也不再增加一组 work-stealing thread。无 policy 的 worker 本地缓存 upstream/config 快照，只在每秒或 reload 后刷新，普通数据报不会抢动态配置锁。新会话 pending 去重是分片集合，association prune 按时间、创建数量和容量压力节流，避免重连风暴每建一条会话就扫全表。

Hooks and plugins

脚本应该放在哪里，不应该替代什么

proxysss 的产品方向很明确：它是通用网关，脚本是扩展点，不是把基础能力重新写一遍的借口。

Hook 适合做什么 不该做什么
access HTTP 路由阶段做小范围业务决策、path 改写、header 注入。 不要拿来重写整个基础反向代理面。
balancer 在已有路由基础上改 upstream、灰度、亲和。 不要在这里做慢逻辑或大量序列化。
preread 给 TCP / UDP 做前置选路。 不要把实时热路径变成脚本业务 worker。
log 观测、计数、轻量记录。 不要做阻塞或高成本计算。 
script:
  args:
    - "./gateway.ts"

plugins:
  auto_load_dir: "./plugins"

这两条配置说明的是“脚本从哪里来”，不是“把所有能力都交给脚本”。固定网关能力优先用 YAML，本体能力缺口再推动 Rust / YAML 面完善。

哪些东西能热重载，哪些一定要重启

这是架构页必须讲明白的运维边界，否则文档再漂亮也不适合真实生产。

可以热重载 需要重启

路由面 `services.reverse_proxy.routes`、`services.service_discovery`、`static_sites`、`webdav`、FTP upstream 管理面与 TLS 关键边界 `admin.bind`、`http.tls.mode`
Warm-up 会重跑 注册中心映射读取、热文件缓存、sendfile 准备、上游 keepalive 预拨、`/healthz` warm 状态 日志核心设置 日志路径、日志级别等核心输出边界
Performance discipline

性能不是单链路冲分，而是全局无副作用优化

这是 proxysss 架构设计里非常核心的一条线：热路径避免全局锁、尽量使用分片或原子结构、静态和流式都要参与 warm-up、结论必须靠 mixed-load 压测证明。

Hot path

不要让控制面锁卡住数据面

热路径优先 per-connection、per-listener、per-worker 分片和池化；全局锁只留给配置、证书、控制面协作。

Warm-up

配置加载后先自我预热

静态热文件、sendfile 描述符、反向代理 keepalive 池都会先准备好，避免首个真实请求承担冷启动成本。

Validation

只赢一项，不算赢

generic SSE、静态、HTTP reverse proxy、WebSocket、TCP、UDP 与透明 QCP 都要一起看；角色隔离脚本的 mixed/equal-load 默认交错跑 4 次并比较中位数，任一轮错误取最大值，equal-offered-load 再严格判 p50/p95/p99。KCP/QCP frame termination 不拿 nginx 做伪对照；QCP 行只验证与等价 nginx UDP listener 的透明 edge forwarding。

Fast lane

默认头语义也要能吃到快路径

plain HTTP 的 raw reverse proxy 和 raw SSE fast lane 会保留 `X-Forwarded-*` / `Forwarded`，不需要为了性能去关闭可观测性或改配置语义。Plain/TLS fast-lane request reader 让 socket 直接读入 persistent per-connection BytesMut 的 spare capacity，并把已经找到的 head_end 返回 caller，去掉每请求的临时 4 KiB 缓冲复制与第二次 CRLF 扫描。同一 keep-alive connection 重复的 exact static GET head 在首次校验后直接命中 connection response cache，直到 revalidation deadline 前不再重复 UTF-8/header parse 与 route lookup；exact raw reverse request 同样复用已验证的 parsed request、完整 serialized upstream request bytes 与已选 route/upstream pool，并在 path 分类前命中，避免重复 static/SSE/WebSocket prefix 判断、route/pool 查找以及 target/path/forwarding header 的 String/Vec allocation。tiny cached static connection 每完成 32 个请求做一次显式 cooperative yield；64 批次会让 static keep-alive 长时间保持 per-core shard runnable，并压低 TLS/realtime/UDP sibling，因此属于 mixed regression boundary。raw reverse 每次请求本身都会等待 upstream write/read 与 downstream write readiness，因此不再叠加重复的周期性 yield。Raw HTTP pool checkout 会 nonblocking 探测并丢弃 EOF/报错/脏 idle socket；预热会同时持有目标连接数后再归还，避免共享起跑屏障后的首个 SSE/reverse 请求命中 stale fd。Raw reverse 让 upstream socket 直接读入 bounded per-connection reusable buffer 的 spare capacity，去掉临时 4 KiB 栈块及复制；小 fixed-length 响应继续在同一 allocation 中解析，并把 raw head + 已到达 body 一次写回，只有尚未收全的 body 才进入 bounded ByteBufferPool relay，避免每响应创建 BytesMut、拆分 Bytes 与二次拼接。4 KiB 以内重复且完全相同的 upstream response head 还会命中 per-downstream framing cache：partial read 先比较 cached prefix，exact head 直接复用 status/body framing，跳过 delimiter search、httparse 与 64-header scan；head 变化时立即完整解析并替换 cache，response body 从不缓存，超出 4 KiB 的 head 也不会常驻 connection memory。

WebSocket relay

游戏重连不会反复打分配器

plain WebSocket 与 WSS 在握手后使用多核 Tokio relay；只有出现 pending 数据时才从有界 `ByteBufferPool` 借 buffer，空闲长连接不会预占每方向固定大块内存。4c 生产参考包络默认验证 20k idle WSS 容量和最多 4096 条活跃消息连接，并分别比较握手/消息的 p50、p95、p99，记录内存 current/peak 和每连接成本；它不是任意固定 RAM 准入门槛。

Static sendfile

零拷贝也要保证并行公平

ALPN 已确认的 HTTP/2 直接进入 Hyper H2 builder，不重复 HTTP/1 协议探测；TLS accept、handshake 与 H2 connection 全程留在独立 crypto runtime 的同一 connection owner，避免中途跨 runtime wake。mixed H2 budget 为每 cpuset CPU 3,800 ops/s，使用旧 2,000 上限留下的 gateway 空闲预算，同时继续给 plain HTTP/UDP/realtime 留出确定份额。`SO_REUSEPORT` shard 数来自进程 cpuset。H2 小静态热对象以 immutable `Bytes` 直接应答，并用 stale-while-revalidate 避免每请求 metadata syscall。大静态文件复用缓存 fd 并走 Linux `sendfile`，单次 chunk 随 traffic profile 自适应为 `small=2 MiB`、`balanced=16 MiB`、`bulk=16 MiB`。默认 balanced 让 accept shard 的固定 connection owner 直接在 Tokio writable readiness 上 drain partial writes，直到 chunk 预算、EOF 或 `EAGAIN`，避免低密度 connection/per-response runtime migration、fd duplication、eventfd、epoll add/del 与 oneshot 往返；socket 在 `EAGAIN` 时自然让出，完整 sendfile 响应后 cooperative yield 一次。为避免高并发 bulk connection 连续占住单线程 owner，同时保留大文件吞吐，每条 connection 用本地 sequence 让每 3 个响应中的 2 个在 8 MiB 中点额外 yield、另 1 个保持完整 16 MiB drain；sequence 初相位由 remote source port 均匀分散到三相，避免 16/32 条连接同时在 8 MiB 形成 scheduler convoy。balanced 在 active large responses 达到 `1 × cpuset cores` 后，把后续 duplicated socket/file fd 动态交给有界、`ceil(cpuset cores / 4)` 分片、round-robin 的原生 epoll sendfile reactor；计数发生在进入当前响应前，因此每核 2 条持续 keep-alive large connection 也会在稳态全部迁出 HTTP owner，不会卡在临界值。sendfile owner 固定在 cpuset 前部；sparse realtime owner 使用 soft CPU ownership，一核一 owner 才固定到 cpuset 后部。HTTP/TCP accept/UDP 共用的 per-core shard 保持 nice 0，TLS crypto runtime 为 nice +3、sendfile owner 为 nice +6、realtime owner 为 nice +5。低于基线门槛继续直送，达到门槛后逐步 offload；阈值与 worker 数都随高核主机线性增长，明确选择 bulk 则始终走全核 nice 0 reactor。密度只在响应边界使用一个 relaxed active counter，chunk 热循环不加锁。`sendfile(2)` 使用每响应独立的显式 offset，handoff 失败则从当前 offset 回退 Tokio readiness。默认 small 仍做 125µs pacing。Range 与 backpressure 语义保持不变，同轮仍必须重新通过全部混合场景。

`benchmark-ubuntu24-amd64-docker.sh` 是禁止 GitHub Actions 性能压测后的本机/原生 Docker 入口：硬校验 controller 与被测镜像为 Ubuntu 24.04 x86_64，在容器内构建当前 checkout，并把 gateway、backend、load client 固定到互不重叠的 cpuset/cgroup。这样更快的 closed-loop TCP/UDP 不会让 client/backend 多消耗 CPU、反向饿死同轮 HTTP。HTTP、HTTPS、static-large、SSE、WebSocket、TCP、UDP 与透明 QCP 按 1×/2×/4× 一起放大；逐档同时判 mixed saturation 吞吐和 equal-offered-load p50/p95/p99，要求零错误、逐场景与聚合严格胜出。equal-load 默认取双方较慢 saturation 的 25%，为 11 个并行 client 保留真实 latency headroom；双方仍须完成至少 98% target。arm64 daemon 明确记录 `execution_mode=emulated-amd64`，不能冒充物理 x86。底层 `benchmark-all-scenarios-isolated.sh` 默认按 4+4+8 CPU 分配，也允许本机 wrapper 按 Docker 可用核数等比例切分，报告从实际 gateway cpuset 计算核数。所有 client 容器先 create、再并行 start，进程先等待共享 volume 上的 release file；确认全部容器都已运行后，控制器才写入统一绝对时间，各进程再把它作为 `--start-at-unix-ms` 交给进程内 worker barrier。测量窗口不会被 Docker create/start（包括 amd64 模拟启动）缩短或错开。nginx 1.31.2 mainline 使用 `-O3 -fno-plt`、相同 gateway cpuset 与等价协议配置。mixed/equal-load 默认交错运行 4 次并取中位数，任一轮错误取最大值；serial isolated saturation 可另跑 3 次做诊断，但不能替代 mixed gate。`benchmark-websocket-production-gate.sh` 以四轮 Latin-square 逐级验证 256/1024/4096 active WSS 与 5k/10k/20k idle WSS。`benchmark-cross-host-wss.sh` 再从独立 client host 对独立 gateway/backend 复跑，并用远端 systemd cgroup 强制 4 CPU、300k nofile，保存同 SHA binary、cgroup current/peak 与每连接内存成本、主机指纹、nginx -V 与原始样本。Docker role isolation 不能冒充三台物理机。

严格矩阵中的 `protocol: qcp` 是透明 UDP edge forwarding：proxysss 与 nginx 等价 UDP listener 接受相同负载和门禁。该结果不代表 proxysss 在热路径解析或终止 QCP frame；KCP 专用封装仍不进入默认 nginx head-to-head 矩阵。
