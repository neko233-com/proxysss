migration / nginx to proxysss

把 nginx 配置迁到 proxysss，不靠猜。

这页的目标不是讲一堆能力名词，而是把常见 `nginx` 配置翻译成适合 `proxysss` 的 YAML。先有能跑的配置，再理解该把能力落在哪个面。

先迁最常见场景 再看高级迁移 回 docs 首页
Migration mindset

先记住该把配置写在哪里

你不用先记所有字段，只要先判断这条能力属于 `reverse_proxy`、`domain_routes`、`static_sites` 还是 `tcp/udp listeners`。

nginx 思路 proxysss 落点 适合什么
`server_name + location /` `services.reverse_proxy.routes` 普通整站反代
`location /api/` + `proxy_pass` `match.path_prefix` + `strip_prefix` API 前缀代理
同域名多个路径拼成整站 `services.domain_routes` 站点级编排
`stream {}` `tcp.listeners` / `udp.listeners` 数据库、游戏、MQTT、UDP
Beginner path

先迁最常见的配置

下面这些场景覆盖了大多数第一次迁移的站点。

API prefix

`location /api/`

services:
  reverse_proxy:
    routes:
      - name: api
        match:
          hosts: ["api.example.com"]
          path_prefix: "/api"
        strip_prefix: true
        upstreams:
          - url: "http://127.0.0.1:8080"
Whole site

整站反代

services:
  reverse_proxy:
    routes:
      - name: app
        match:
          hosts: ["app.example.com"]
          path_prefix: "/"
        upstreams:
          - url: "http://127.0.0.1:9000"
Static site

静态站点

services:
  static_sites:
    - name: homepage
      hosts: ["www.example.com"]
      root_dir: "/srv/www"
      index_files: ["index.html"]
Auto HTTPS

自动 HTTPS

http:
  plain_bind: 0.0.0.0:80
  tls_bind: 0.0.0.0:443
  tls:
    mode: acme_managed
    acme:
      enabled: true
      email: "ops@example.com"
      challenge: http01
Advanced path

高手迁移：从普通 HTTP 走向完整网关

迁移不是只翻译语法，更是把能力放到更合适的网关面。

Domain routes

站点级编排

services:
  domain_routes:
    - domain: "example.com"
      routes:
        - path_prefix: "/"
          service: static:homepage
        - path_prefix: "/api"
          service: reverse_proxy:api
Load balance

集群 + health

load_balance:
  strategy: weighted_round_robin
  active_health:
    path: "/healthz"
    interval_secs: 5
    timeout_ms: 1200
AI / SSE

流式 AI 代理

runtime:
  performance:
    enabled: true
    traffic_profile: small

services:
  ai_proxy:
    routes:
      - name: llm-edge
        transport:
          flush_interval_ms: 0
          tcp_nodelay: true
TCP / UDP

`stream` 迁移

tcp:
  listeners:
    - name: postgres
      bind: 0.0.0.0:5432

udp:
  listeners:
    - name: realtime
      bind: 0.0.0.0:7001
      session_ttl_secs: 45
      max_associations: 150000
Validation

迁移验证不是看一张 benchmark 图

真正的迁移完成，应该同时确认配置面、路由视图、能力面和 mixed-load 性能纪律。

proxysss tune linux --apply
proxysss config explain
proxysss config routes
proxysss config nginx-parity --format yaml
scripts/benchmark-all-scenarios.sh

默认 GitHub Actions CI 现在只负责全平台二进制打包，不自动跑 benchmark；迁移性能判断需要手动/专机跑上面的 mixed-load。v1.3.5 UDP fast path 当前专项结果是 `udp-stream 4.045x`，旧历史 mixed-load UDP 行不要当成当前结论。

迁移原则： 先把一条 `reverse_proxy.routes` 跑通，再拆 `static_sites`、`ai_proxy`、`tcp.listeners`、`udp.listeners`，最后补 TLS、health、cache 和 benchmark。这样最容易发现副作用回归。
