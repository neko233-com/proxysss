configuration / human guide

配置页先解决“该配哪里”，不是先把字段全砸给你。

proxysss 的 YAML 目标不是重写一份 nginx 指令百科，而是让你能更快把 HTTP、HTTPS、AI SSE、静态站点、TCP、UDP、MQTT/IoT 边缘流量和自动化管理面放到正确的配置面里。

Beginner path

先把一条工作链路跑通，再学抽象。

这页先给你生产形态的最小配置，再解释为什么它应该落在这个能力面，而不是别处。

第一个站点

最小反向代理

适合先把 `app.example.com` 代理到本机服务，确认域名、监听端口和上游路径都正常。

http:
  plain_bind: 0.0.0.0:80

services:
  reverse_proxy:
    routes:
      - name: app
        match:
          hosts: ["app.example.com"]
          path_prefix: "/"
        upstreams:
          - url: "http://127.0.0.1:9000"
正式上线

HTTP 升 HTTPS

如果 80/443 能直连公网，默认优先走内建 ACME。这样证书生命周期、reload 和管理面都还在同一套产品节奏里。

http:
  plain_bind: 0.0.0.0:80
  tls_bind: 0.0.0.0:443
  tls:
    mode: acme_managed
    acme:
      enabled: true
      email: "ops@example.com"
      challenge: http01
官网 + API

静态站点和 API 分开声明

静态文件适合单独走 `static_sites`，API 保持在 `reverse_proxy.routes`。HTML/CSS/JS/图片/字体/音视频和大文件下载支持 Range 断点续传，这样排障、缓存和性能预热都更清楚。

services:
  static_sites:
    - name: homepage
      hosts: ["www.example.com"]
      root_dir: "./www"
      index_files: ["index.html"]

  reverse_proxy:
    routes:
      - name: api
        match:
          hosts: ["api.example.com"]
          path_prefix: "/"
        upstreams:
          - url: "http://127.0.0.1:8080"
Choose the surface

大部分配置问题，真正的核心是“你选错层了”。

proxysss 把通用网关能力拆成多个清晰配置面。先选对面，再谈字段。

配置面 适用场景 不要误用成
services.reverse_proxy.routes
最常用 HTTP 代理面		 网站、API、后台系统、SSE、WebSocket、HTTP/2 gRPC、下载站。 不要把所有业务逻辑都塞进脚本替代它。
services.domain_routes
站点总编排		 一个域名下有静态、API、WebDAV 等多个服务，需要统一入口视图。 不要把它理解成唯一实际执行面。
services.ai_proxy
AI / SSE / OpenAI-compatible		 New API、OpenAI-compatible、SSE 流式输出、长响应代理。 不要把所有 HTTP 请求都默认塞进 AI proxy。
services.static_sites
静态文件分发		 官网、文档站、资源分发、CDN 回源、预热缓存、404 页面、Range 下载。 不要为了分发静态资源先写一层脚本。
services.service_discovery
Consul / etcd / Nacos		 把注册中心服务映射到 HTTP、TCP、UDP 上游池的控制面声明。 不要让热路径每个请求都查注册中心。
tcp.listeners
TCP 原生流量		 数据库、游戏长连接、MQTT TCP、TLS passthrough/SNI。 不要期待这里能做 HTTP 头级别策略。
udp.listeners
UDP、KCP、QCP、CoAP-style		 实时设备、游戏、会话型 UDP 网关。 不要把它包装成协议终端本身。
Common combinations

高手最常用的组合不是“更复杂”，而是“边界更清楚”。

这些组合都围绕真实上线情境写：缓存、健康检查、DNS-01、管理自动化、游戏长连接和 UDP 并行接入。

API 集群

反向代理 + 权重 + 健康检查 + 缓存 + 限流

services:
  reverse_proxy:
    routes:
      - name: api-cluster
        match:
          hosts: ["api.example.com"]
          path_prefix: "/"
        upstreams:
          - url: "http://10.0.0.11:8080"
            weight: 3
          - url: "http://10.0.0.12:8080"
            weight: 2
        load_balance:
          strategy: weighted_round_robin
          active_health:
            interval_secs: 5
            timeout_ms: 1200
            path: "/healthz"
        cache:
          strategy: respect_origin
          stale_while_revalidate_secs: 30
          stale_if_error_secs: 120
        rate_limit:
          requests_per_second: 200
          burst: 400
Wildcard TLS

泛域名证书优先走内建 DNS-01

http:
  tls_bind: 0.0.0.0:443
  tls:
    mode: acme_managed
    acme:
      enabled: true
      email: "ops@example.com"
      challenge: dns01
      dns:
        provider: cloudflare
        credentials:
          api_token: "REDACTED"

Game + UDP

游戏长连接和 UDP 同时接入

tcp:
  listeners:
    - name: game-tcp
      bind: 0.0.0.0:7000
      nodelay: true
      connect_timeout_ms: 1500

udp:
  listeners:
    - name: game-udp
      bind: 0.0.0.0:7001
      session_ttl_secs: 45
      max_associations: 150000
Reload + warm-up

热重载不是“只会重新读 YAML”，还会做预热。

proxysss 在配置加载和热重载后会重新计算路由，读取 service_discovery 映射，预热静态文件缓存或 sendfile 描述符，并为 reverse proxy / AI proxy 预拨 keepalive 池。

可热重载
  • 大部分合并后的配置值
  • `proxysss.yaml`
  • 主脚本与自动加载插件脚本
  • `services.reverse_proxy.routes`、`services.service_discovery`、`static_sites`、`webdav`、FTP upstream
需要重启
  • `http.plain_bind`、`http.tls_bind`、`http.h3_bind`
  • `admin.bind`、TCP / UDP listener bind 集合
  • `http.tls.mode`
  • 日志路径与核心日志级别设置
Performance

性能结论必须看 mixed-load，不看单场景截图。

proxysss 的性能优化必须是无副作用优化。generic SSE、静态、HTTP reverse proxy、WebSocket、TCP、UDP 要一起看；KCP/QCP 作为独立 UDP listener 能力保留，但不进入当前性能 benchmark 矩阵。

先调 Linux 
proxysss tune linux --apply
proxysss config explain
proxysss config capabilities
proxysss config reload-plan
scripts/verify-docker-scenarios.sh
.\scripts\verify-docker-scenarios.ps1
scripts/benchmark-all-scenarios.sh

默认 GitHub Actions CI 现在只负责全平台二进制打包，不自动跑测试、smoke 或性能压测。Docker 场景验证是手动入口，用于检查全场景样例、Range、注册中心配置和能力矩阵；benchmark 是手动/专机性能验证入口。v1.3.5 UDP fast path 当前专项结果是 `udp-stream 4.045x`，`proxysss 127742.75 ops/s` vs `nginx 31577.33 ops/s`，两边 0 错误。

traffic_profile

`small` 适合小文件、SSE、HTTP2、小包 flush；`balanced` 是默认推荐；`bulk` 适合明显偏大文件和下载分发。

runtime:
  performance:
    enabled: true
    traffic_profile: small

Mistakes

最常见的三个方向性错误

这些不是单个字段的拼写错误，而是会把整个网关越配越乱的思路问题。

误区 1

把所有问题都塞进脚本

脚本适合业务定制，不适合替代 HTTP、TLS、缓存、限流、路由和流量治理这些原生能力面。

误区 2

拿单条 SSE benchmark 当发布结论

项目要求已经明确：后续所有性能优化都要压测，而且要无副作用。只赢单路链路不算赢。

误区 3

一上来就开公网写管理面

默认的 loopback + `enable_write_ops=false` 是刻意设计的安全姿势。自动化需要时再显式开启。
