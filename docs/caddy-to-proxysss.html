migration / caddy to proxysss

把 Caddyfile 迁到 proxysss，同时看清能力面。

从 `caddy` 迁到 `proxysss`，不只是改语法。你会从“很顺手的站点代理”切到“更完整的通用网关”，所以文档会一边给你直接可抄的 YAML，一边告诉你哪些能力应该拆到独立配置面。

先迁最常见场景 再看高级迁移 回 docs 首页
Migration mindset

先理解 Caddy 和 proxysss 的思路差别

`Caddy` 的强项是上手快和自动 HTTPS；`proxysss` 的目标是把 nginx 级网关能力放进一套更容易维护的单文件 YAML。

Caddy 思路 proxysss 落点 适合什么
`reverse_proxy` `services.reverse_proxy.routes` 普通站点反代
`handle_path /api/*` `match.path_prefix` + `strip_prefix` API 前缀改写
`file_server` `services.static_sites` 静态站点
自动 HTTPS `http.tls.mode: acme_managed` 默认推荐
`layer4` 场景 `tcp.listeners` / `udp.listeners` 原生 TCP / UDP
Beginner path

先迁最常见的站点代理

这些例子适合从 Caddy 最常见的使用方式平滑切过来。

Whole site

整站反代

services:
  reverse_proxy:
    routes:
      - name: app
        match:
          hosts: ["app.example.com"]
          path_prefix: "/"
        upstreams:
          - url: "http://127.0.0.1:9000"
API path

`handle_path /api/*`

services:
  reverse_proxy:
    routes:
      - name: api
        match:
          hosts: ["app.example.com"]
          path_prefix: "/api"
        strip_prefix: true
        upstreams:
          - url: "http://127.0.0.1:8080"
Static site

`file_server`

services:
  static_sites:
    - name: homepage
      hosts: ["www.example.com"]
      root_dir: "/srv/www"
      index_files: ["index.html"]
Auto HTTPS

自动 HTTPS / WSS

http:
  plain_bind: 0.0.0.0:80
  tls_bind: 0.0.0.0:443
  tls:
    auto_https:
      domains: [wss.example.com]

只给域名即可走内建 managed ACME 的正式 TLS-ALPN-01，无需 certbot、acme.sh、DNS API 或邮箱。A/AAAA 必须指向网关，且公网 443 可达；邮箱只是到期和安全通知的可选项。旧的显式 HTTP-01（需 80）/ TLS-ALPN-01 / DNS-01 配置保持兼容。

Advanced path

从站点代理走向完整网关

这是 Caddy 用户最容易低估的一步：proxysss 不是另一个语法不同的 `Caddyfile`，而是更完整的网关面。

Load balance

负载均衡 + health

upstreams:
  - url: "http://10.0.0.11:8080"
    weight: 3
  - url: "http://10.0.0.12:8080"
    weight: 2
load_balance:
  strategy: weighted_round_robin
  active_health:
    path: "/healthz"
    interval_secs: 5
    timeout_ms: 1200
AI / SSE

流式 AI 代理

runtime:
  performance:
    enabled: true
    traffic_profile: small

services:
  ai_proxy:
    routes:
      - name: llm-edge
        transport:
          flush_interval_ms: 0
          tcp_nodelay: true
TCP / UDP

`layer4` 迁移

tcp:
  listeners:
    - name: mqtt-tcp
      bind: 0.0.0.0:1883

udp:
  listeners:
    - name: coap
      bind: 0.0.0.0:5683
      session_ttl_secs: 30
      max_associations: 50000
WebDAV

文件协作入口

services:
  webdav:
    - name: assets-dav
      hosts: ["dav.example.com"]
      path_prefix: "/"
      root_dir: "./dav-data"
      read_only: false
Validation

迁移完成的标准，要看 mixed-load

proxysss 的迁移和后续性能优化，不能只看单站点 HTTP。generic SSE、static、HTTP reverse proxy、TCP、UDP 要一起看，而且必须无副作用；KCP/QCP 是独立能力，不进入当前性能 benchmark 矩阵。涉及公网/NIC 延迟时，必须从独立 client 主机运行 scripts/benchmark-cross-host-wss.sh，Docker cpuset 数据不能冒充跨机结果。

proxysss tune linux --apply
proxysss config explain
proxysss config capabilities
proxysss config reload-plan
scripts/benchmark-all-scenarios.sh

默认 GitHub Actions CI 现在只负责全平台二进制打包，不自动跑 benchmark；迁移后的性能判断需要手动/专机跑上面的 mixed-load。v1.3.5 UDP fast path 当前专项结果是 `udp-stream 4.045x`。

迁移原则： 先迁最简单的 `reverse_proxy`，再拆 `static_sites`、`ai_proxy`、`tcp.listeners`、`udp.listeners`，最后补 TLS、health、cache 和 benchmark。这样最容易看清能力边界，也最容易发现副作用。
